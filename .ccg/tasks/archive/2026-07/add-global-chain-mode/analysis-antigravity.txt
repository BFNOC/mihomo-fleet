根据对 `mihomo-fleet` 代码库的只读分析，针对新增“全局链式代理”模式的需求，以下是详细的分析报告与架构建议：

### 1. 建议的数据模型/API/UI 变更范围 (Info)

*   **数据模型 (`types.go`)**：
    *   在 `Instance` 和 `InstanceView` 结构体中新增 `ProxyMode string`（用于区分当前 `rule` 模式和新增的 `global-chain` 模式）。
    *   新增 `LocalProxies string` (YAML格式) 或 `[]map[string]any`，用于存储用户在此实例中本地添加的节点。
    *   新增 `ChainDirection string`（可选），用于决定本地节点是作为前置代理还是后置代理（即 `relay` 数组的顺序）。
*   **API 接口 (`controller.go`)**：
    *   `/api/instances` (POST, PUT) 需要接收并处理上述新增的 `ProxyMode`、`LocalProxies` 等字段，写入持久化存储。
*   **UI 交互 (`web/app.js` & `index.html`)**：
    *   **新建/编辑面板**：增加“实例模式”下拉框。选择“全局链式代理”后，动态展示一个用于输入“本地附加节点”的 YAML 多行文本框。
    *   **节点面板**：对于新模式，UI 应该只展示代理链相关的节点组（详见下方兼容性风险），供用户在订阅池中单选节点。

### 2. runtime config 生成方案 (Info)

在 `config.go` 的 `writeRuntimeConfig` 函数中，需要根据 `ProxyMode` 执行不同的配置生成分支。对于新模式：

1.  **节点保留与提取**：
    *   解析并提取 Profile 配置中的 `proxies` 和 `proxy-providers` 原封不动地保留。
    *   **丢弃原分流**：彻底删除解析出的 `proxy-groups`、`rules`、`sub-rules`，抹除订阅自带的分流逻辑。
2.  **构造节点池组 (Subscription Select)**：
    *   自动生成一个名为 `SubscriptionNodes` 的 `type: select` 代理组。
    *   将其 `proxies` 字段填入所有提取出的单个节点名。
    *   将其 `use` 字段填入所有提取出的 `proxy-providers` 名字。
3.  **构造链式代理组 (Chain Relay)**：
    *   将用户填写的 `LocalProxies` 解析并追加到配置的 `proxies` 列表中。
    *   自动生成一个名为 `GlobalChain` 的 `type: relay` 代理组。
    *   其 `proxies` 数组按用户的链式顺序，填入本地节点名和上一步的 `SubscriptionNodes`。
4.  **强制全局路由**：
    *   配置项固定为 `mode: rule`。
    *   重写路由表，仅保留一条：`rules: ["MATCH,GlobalChain"]`。

### 3. 兼容性和迁移风险 (Critical & Warning)

*   **[Critical] 代理组内存状态与运行状态严重脱节风险**：
    *   **现状**：当前 `/api/profiles/:id/proxies` 接口以及 `store.go` 中的 `reconcileInstanceSelection` 函数，**全都是直接解析原始的 Profile 配置文件 (`parseProfileProxyGroups`)** 来向 UI 输出节点组列表的。
    *   **风险**：由于新模式下 `writeRuntimeConfig` 会在运行时魔改、丢弃原始组并生成 `SubscriptionNodes`，如果 API 不改，UI 将依然展示订阅里原有的分组。用户点击 UI 切换节点时，Mihomo API 将报错找不到对应分组。
    *   **解决方案**：必须修改 `/api/profiles/:id/proxies` 接口逻辑。如果查询携带了 `instanceId` 且该实例是 `global-chain` 模式，应解析其对应的 Runtime Config（或在内存中模拟生成后的 config），将真实的 `SubscriptionNodes` 返回给 UI。
*   **[Warning] 数据向下兼容风险**：
    已有的所有 Instance 数据没有 `ProxyMode` 字段。在 JSON 反序列化后为空字符串，业务逻辑中必须将空值、`rule` 等同于现有的“常规分流模式”，以确保用户的存量实例无缝升级不受影响。
*   **[Warning] 订阅节点为空的边界情况**：
    某些订阅可能只使用了 `proxy-providers` 而 `proxies` 数组完全为空。在构造 `SubscriptionNodes` 时，必须做好空数组判空和兼容，防止 Mihomo 启动解析报错。

### 4. 必须补的测试 (Info)

*   **配置生成隔离测试 (Unit Test)**：
    针对 `config.go`，传入含有各种复杂 `rules` 和嵌套 `proxy-groups` 的模拟订阅配置，断言在 `global-chain` 模式下生成的 YAML：是否完全没有原 `rules`，是否正确包含 `type: relay` 和 `MATCH` 规则，且本地 proxy 是否被正确并入。
*   **节点选择校正测试 (`reconcileInstanceSelection`)**：
    模拟一个实例从“普通模式”切换为“链式模式”，验证 Store 能够正确丢弃原本绑定在旧订阅分组上的失效节点选择，避免残留的僵尸选择在 Mihomo 启动后不断重试。
*   **动态代理组解析测试 (`parseProfileProxyGroups`)**：
    验证该函数（或新重构的对应函数）在针对新模式进行解析时，能否正确吐出给前端使用的 `SubscriptionNodes` 组，并且屏蔽被抛弃的原生组。
